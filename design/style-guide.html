19 Do's and Don'ts

✓ Do
  • Use CSS variables for every color and shadow
  • Use the defined spacing scale
  • Navy headers on cards, panels, and modals
  • Orange for primary actions, focus rings, selected states
  • System font stack — no Google Fonts
  • transition: var(--transition) on interactive elements
  • cursor: not-allowed + opacity: 0.6 for disabled
✗ Don't
  • Invent new colors outside the palette
  • Use inline styles (except for dynamic JS values)
  • Shadows heavier than --shadow-lg
  • Border-radius >10px or <4px
  • Center-align body text in data-heavy views
  • More than 3 font sizes per component
  • External CSS frameworks (Bootstrap, Tailwind…)

20 Potential Future Changes

Token System: --color-gray and --color-border are identical (#e0e0e0) — one is redundant. Hardcoded RGBA values (rgba(238,127,0,…)) and #856404 leak through without tokens.
Spacing contradiction: Scale says "no in-between values" but sections 7/8/9 use 0.6rem, 0.85rem, 0.78rem — all off-scale.
Typography gaps: No line-height defined. Sizes 0.78rem, 0.9rem, 1.75rem used in components but missing from the scale table.
Naming contradiction: Convention says component-element--modifier (double dash) but every example uses single dashes. Unprefixed state classes (.error, .selected) risk collisions.
Missing patterns: Z-index scale, empty states, responsive split panel, checkbox/radio inputs, modal close button style, textarea resize guidance.
Mixed units: Section 4c sidebar uses padding: 16px (pixels) while everything else uses rem.
